About User Stories

A User Story is a lightweight requirements artifact that works well when using a collaborative approach to application software development. It describes an interaction between an actor and a software system. The general guidelines are:

  1. A User Story is brief and not too detailed. It serves as a placeholder for a conversation. When working in a collaborative style, the assumption is that interested parties will interact directly with one another to achieve the project goals. User Stories represent work to be done, and they exist only until the work they represent has been done; they are transient artifacts, and not a permanent record of system requirements or development history. Details about the requirements may be stored in a long-lived system of record, separate from the User Stories. When we do the work represented by a User Story, we say we are "playing" the story.

  2. A User Story captures four key pieces of information:
    • What is the desired functionality?
    • Who needs the desired functionality?
    • What is the value to be obtained through the desired functionality?
    • How will we know when the desired functionality has been completed?

Format

There is no single "correct" format for a User Story. People who are new to User Stories often use the following template to help them get started. Over time, they may adjust the format according to their own needs. 

As a [persona or user type]                         <= Who needs the desired functionality?
I want [something to happen]                        <= What is the desired functionality?
So that [I will receive such-and-such value]        <= What is the value to be obtained?

Acceptance Criteria:                                <= How will we know when we are done?

1. Demonstrate that [something happens] by showing the working software in action
2. Demonstrate that [whatever else is relevant to completing the User Story] has happened

This might look like: 

As a Cashier,
I want the Register to calculate the total sale amount
So that I can charge the Customer the correct amount

Acceptance Criteria:

GIVEN the customer buys 2 bags of potato chips 
AND 1 soft drink
WHEN I complete the sale
THEN the total price is $4.79

Details such as the unit price of the items, whether they are taxable, and the tax rate to apply are not spelled out in the User Story. They are documented elsewhere and/or they emerge through collaboration among project stakeholders. In addition, the User Story focuses on the actions taken and the visible outcomes of those actions, and not on implementation details like "swipe a credit card" or "click a button." Notice the sample story does not specify how the customer "buys" an item or how the cashier "completes the sale."

INVEST Principles

We usually start a development project with a roadmap or plan that lays out the order in which features will be delivered. However, reality often intrudes on this plan and causes us to change it. User Stories might be shifted around for a variety of reasons — changes in business priorities, changes in regulatory requirements, technical prerequisites discovered after the project starts, resource availability (or lack thereof), dependencies on external suppliers, or changes to the design due to lessons learned along the way.

To make it as easy as possible to shift stories around, we try to keep them small and minimize the dependencies between them. Bill Wake coined the acronym INVEST to help us remember key principles for good User Stories.

  • Independent. A User Story does not depend on any other User Story. The stories can be played in any order.

  • Negotiable. Stakeholders and team members discuss the details of the User Story and reach agreement on how the functionality will be implemented, demonstrated, and delivered.

  • Valuable. A User Story represents functionality that has value to at least one stakeholder. It does not represent an activity that is disconnected from stakeholder value.

  • Estimatable. A User Story is not ready to be played if the team has insufficient information to be able to judge the level of effort.

  • Small. Small is a relative term, and its meaning depends on the nature of the work you are doing and on the delivery time you have to meet. In general, it's best to break functionality down into the smallest vertical slices that can be demonstrated to a business stakeholder.

    • Testable. It's hard to prove the functionality works if there's no practical way to test it. A User Story that is ready to play includes acceptance criteria that are clear and that can be tested unambiguously.
